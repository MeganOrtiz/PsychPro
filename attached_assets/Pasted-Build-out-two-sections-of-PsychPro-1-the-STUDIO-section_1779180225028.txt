Part 3 — Connections feature in CONNECT section
Add a "Connections" tab as a sub-item under the CONNECT section header in the sidebar. (CONNECT also contains the existing Feedback tab.)
Connections page layout:

Top section: "People you might want to connect with"
Card grid of suggested users based on overlapping interest tags
Each card shows: profile photo/initials, display name, role, institution (if filled in), top 3 shared interest tags (highlighted as "shared with you"), bio preview
Each card has one CTA button: "Request introduction"

Matching logic:

Only show users who have BOTH "Allow connection suggestions" AND "Open to networking introductions" toggles ON
Sort suggestions by number of overlapping interest tags (descending), then by recency of profile activity
Exclude users the current user has already connected with or declined
Show max 12 suggestions per page with "Show more" pagination

Double opt-in flow:

User A clicks "Request introduction" on User B's card
System creates a connection_request record (status: pending)
User A sees a confirmation: "We've sent your interest to [User B]. If they're also interested, we'll connect you both via email."
User B receives an in-app notification AND an email: "A fellow PsychPro user is interested in connecting based on your shared interest in [top 3 shared tags]. View their profile and decide if you'd like to connect."
User B can view A's profile and click Accept or Decline
On Accept: system sends a single intro email to BOTH users, CC'd, containing each user's display name, role, institution, bio, and shared interests. Subject line: "PsychPro intro: [User A name] ↔ [User B name]". Email body explains they both opted in based on shared interests and lists the shared tags. After this, PsychPro is out of the loop — conversation moves to email.
On Decline: request is silently closed; User A is not notified that they were declined (just doesn't hear back). User B is not shown User A as a suggestion again.

Important note on email infrastructure: the user has NOT yet set up Resend or another email service. For now, implement all email-sending logic but route emails to the server console / logs instead of actually sending. Add a clearly marked TODO: connect Resend or another email service comment at each send point. When email is wired up later, only that integration layer should need to change.
Privacy and abuse safeguards:

Email addresses are NEVER shown in the app or in the suggestion UI — they're only revealed in the intro email after mutual accept
Users can disable connection suggestions at any time from their profile
Add a "Block this user" option on the request notification (in case of abuse) — blocked users never appear as suggestions for the blocker again
Admin sees a "Connection Requests" tab in the ADMIN section showing aggregate stats (sent, accepted, declined, blocked) — but admin should NOT see message content or individual user identities in this report unless investigating abuse
Rate limit: each user can send max 5 outgoing connection requests per week to prevent spam

Database schema for connection_requests:
id, requester_id, recipient_id, status (pending/accepted/declined/blocked),
shared_tags (array, captured at request time),
created_at, responded_at (nullable), intro_email_sent_at (nullable)
