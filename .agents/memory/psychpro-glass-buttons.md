**Rule:** The single source of truth for the glass button look in `artifacts/neuronotes` is `.btn-glass-strong` / `.btn-glass` (defined in `src/index.css`), wired into the shared `<Button>` component (`src/components/ui/button.tsx`) via the variant map. Do not introduce a parallel `.glass-button`, `.cta-glass`, or similar one-off class.

**Current direction = BRIGHT CERULEAN GLOW (reverted from neutral frost).** Owner's latest button direction wants buttons clearly visible on the dark study surface with a cerulean `#76E4F7` glow that *intensifies* idle → :hover → :active (square corners only, no pills). This REVERSES the earlier "neutral frost, no cyan" comp (IMG_2548). The three recipes carrying the look: (1) `.btn-glass-strong` (primary) + `.btn-glass` (secondary/outline) in `src/index.css`, (2) legacy study-page `button.bg-primary`/`button.bg-secondary` + outline `border-input` rules (need explicit `:active` rules too), (3) keep `#76E4F7` only — NEVER mint. **Why:** owner flip-flopped (cyan → neutral frost → back to cyan glow); always trust the most recent task spec over this note. Verify auth-gated buttons via an isolated mockup-sandbox copy mirroring the exact index.css values across idle/hover/active (external Clerk blocks the test browser; a static screenshot can't trigger `:hover`/`:active`, so render explicit per-state sample buttons with the recipe values applied inline).

**SUPERSEDED — current standard is ROUNDED 14px outlined-glow.** The "square / border-radius:0" direction below is no longer in effect. Owner approved (inspired by an ElevenLabs ad) a glowing-OUTLINE button vibe as the shared app-wide standard: a lit 1.5px cerulean `#76E4F7` edge + soft multi-layer halo (idle→hover→active intensifying) + 14px radius + surface-safe light cerulean tint fill (NOT a dark fill — buttons sit on both the dark nebula AND white StudySurface cards). This is carried by the `.btn-glass*` family in `index.css` AND the legacy study-page `button.bg-primary`/`button.bg-secondary`/`button.border.border-input` recipes (keep both in lockstep). Destructive stays near-opaque red, only gains the 14px radius. **Why:** owner loved the landing `.landing-cta-primary` look and asked to roll it out everywhere.

**Pill→square audit rule (HISTORICAL, superseded by the rounded-14px standard above):** Owner's "square" = `rounded-none` / `border-radius: 0` (true 0px corners, NOT `rounded-lg` — that was an older, superseded interpretation). Square EVERY interactive button: shared `<Button>` cva base+sm+lg, `.btn-glass*`, study-page `button.bg-*`, landing `const styles` button classes (`.landing-nav-*`, `.landing-cta`), icon-only triggers, selector/segmented tiles, and Brain Lab hotspot pins. **Exceptions that STAY rounded:** (1) non-interactive elements — avatars, status dots, progress bars/rings/tracks (incl landing `.landing-dash-track/-fill` 999px), count badges, icon containers, `<span>` label chips; (2) content-display card-buttons (e.g. `featured-work` gallery tile, `study-surface` tiles) — they read as cards, not actions; (3) input-like surfaces — file-upload dropzone `<label>`s, inputs/selects/textareas. **Gotcha:** a segmented-control wrapper `<div>` with `overflow-hidden` MUST be squared too, or it clips its inner square buttons back into rounded outer corners. **Audit:** multiline `rg -U "<button(?:[^>]|\n)*?rounded-(full|lg|md|xl|2xl|3xl)"` (className sits on its own line); exclude `components/ui/*` library primitives.

**Why:** An older `.glass-button` utility lived in `src/index.css` and was hardcoded across pages (notably `pages/dashboard.tsx`). When the shared Button was switched to `.btn-glass-strong`, those hardcoded usages kept rendering with the older, milder style because both classes have equal specificity and the cascade order favored the older one. Result: the landing CTAs looked correct but in-app buttons looked flat — and the discrepancy wasn't caught because nobody grepped for the competing class.

**How to apply:**
- When restyling buttons, grep `rg "glass-button|cta-|btn-"` first to surface every competing class.
- Prefer dropping `className` overrides on `<Button>` — let the variant carry the style.
- If a page needs a custom button look, extend the variant map in `button.tsx`, don't add a parallel utility class.

**Sidebar nav glass gotcha:** A "glass/frosted" surface needs a translucent **background fill** (`bg-white/[0.06]`-ish), not just `backdrop-blur` + border + shadow. The sidebar nav tiles (`NAV_ITEM_IDLE` in `src/components/layout/app-layout.tsx`) had blur+border+shadow but no idle fill, so they read as merely *outlined* and the user repeatedly reported them as "not glass/opaque." Adding an idle `bg-white/[…]` fill (matching the Tools Studio tile which already had one) fixes it. **Why:** blur over a dark sidebar with nothing behind the tile produces almost no visible frosting.

**Spotlight "see-through smoke" is a painted image, NOT backdrop-blur.** The dashboard SpotlightCard's smoky look is achieved by painting the same `@/assets/bg/brain-clouds.png` inside the surface with a dark gradient overlay over the StudySurface "dark" base — there is no real translucency to the page behind it. **How to apply:** to make any panel (e.g. the sidebar `<aside>`) "show the smoke through" the same way, layer that image into its `background` (gradient-overlay → `url(smokeBg)` → base gradient), don't reach for translucent fill + `backdrop-filter`.

**All sidebar buttons share one glass helper.** Every clickable row in the sidebar — main nav, Tools Studio tiers, and the profile link — routes through `navItemClass(isActive)` (= `NAV_ITEM_BASE` + `NAV_ITEM_IDLE`/`NAV_ITEM_ACTIVE`). When restyling sidebar buttons, change that helper / the `.nav-glass-*` rules once rather than re-styling individual rows, and don't reintroduce per-row one-off `bg-white/[…]` styles.
